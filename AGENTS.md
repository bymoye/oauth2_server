# 项目协作规则

本文件适用于本目录及其子目录内的所有代码、配置、迁移、测试和文档修改。若用户指令与本文件冲突，以用户当前明确指令为准；若上级系统指令与本文件冲突，以上级系统指令为准。

## 基本原则

- 所有实现遵从第一性原则：先确认问题本质、输入输出、状态来源、边界条件和失败场景，再修改代码。
- 有问题则解决问题本身；需要修改的位置应直接修改，不通过额外中间层、兜底逻辑或兼容包装掩盖错误。
- 不为规避设计缺陷增加“遮羞布”式 fallback。若发现事实源、协议边界、生命周期或数据模型错误，应修正真实根因。
- 尽可能不遗留技术债务。若受外部条件限制无法一次完成，必须把剩余风险写清楚，并避免把临时方案伪装成最终设计。
- 默认以生产级质量处理所有改动；不以“本地”“临时”“测试”为理由降低安全、正确性或可维护性要求。

## 代码设计

- 禁止过度设计、过度抽象和为抽象而抽象。
- 保留最小实现：只写当前问题真正需要的代码，不提前构建未来可能用到的框架、扩展点或适配层。
- 禁止冗余代码。新增代码前先检查现有模块是否已经有同等职责；重复逻辑应收敛到已有职责边界内。
- 文件保持单一职责。不要把协议校验、数据库访问、HTTP 输入输出、配置解析、视图构造混写到同一个文件中。
- 命名应表达领域事实，而不是实现过程、历史来源或临时状态。
- 注释只解释必要的非显然约束、协议原因或安全边界；不要用注释复述代码。

## 修改方式

- 修改前先阅读相关实现、数据结构、迁移和测试，不凭接口名称或局部代码猜测系统行为。
- 修改点应尽量靠近问题发生的位置。只有在重复逻辑已经造成真实复杂度或一致性风险时，才抽取共享函数。
- 不新增无实际价值的兼容层、代理层、适配器、包装器或配置开关。
- 不保留废弃路径、历史实现、无调用代码、注释掉的代码或未使用配置。
- 不引入与当前问题无关的重构、格式化 churn 或依赖变化。

## OAuth / 安全边界

- OAuth、OIDC、DPoP、JWT、Cookie、CSRF、邮件验证、密码哈希和数据库凭据相关改动必须优先保证协议正确性与安全边界。
- 不降低 client authentication、redirect URI、PKCE、refresh token rotation、token revocation、scope、audience、issuer、nonce 等约束。
- 不把客户端输入、外部配置、请求头、Cookie、表单字段或数据库内容视为可信数据。
- 错误响应应符合协议语义，不通过含糊错误码隐藏服务端逻辑问题。
- 敏感响应和凭据材料不得进入日志、README 示例、测试快照或错误详情。

## 依赖与技术栈

- 项目应保持在当前稳定的最佳实践上。Rust edition、crate、工具链、基础镜像、数据库 SDK 和运行时依赖应定期核对最新稳定版本。
- 可以使用新的稳定技术栈，但必须服务于当前问题，不为“更新”本身引入复杂度。
- 升级依赖前应确认 changelog、破坏性变更、安全公告和锁文件变化；升级后必须运行格式化、静态检查和测试。
- 不引入未维护、冷门、职责过宽或会显著增加供应链风险的依赖。
- 若已有标准库、现有 crate 或项目内简单函数可以清晰解决问题，不新增依赖。

## 验证要求

- Rust 代码修改后默认运行：

```powershell
cargo fmt --check
cargo check
cargo clippy -- -D warnings
cargo test --locked
```

- 若某项验证因本机环境失败，应记录具体失败原因，不把未验证描述为已通过。
- 涉及数据库、迁移、OAuth 流程或安全边界的修改，应补充能够证明关键行为的测试。
- 测试应覆盖真实风险点，不为覆盖率制造无意义断言。

## 文档

- README 和项目文档应面向项目本身，不写本次会话、个人环境、临时目录、历史迁移过程或旧实现对比。
- 文档中的能力声明必须与代码真实行为一致。未实现、半实现或依赖外部条件的能力不得写成已完整支持。
- 配置、命令和接口说明应保持中性、准确、可复现。
